## 1. Purpose

This playbook establishes the operational framework governing how the Security Operations Center detects, investigates, contains, and resolves security incidents using Palo Alto Cortex XSIAM. It defines accountability at every tier, sets measurable SLA targets, and creates a repeatable system built to withstand audit scrutiny.

This is a living document. It is updated based on operational findings, post-incident reviews, and platform changes.

---

## 2. Scope

This playbook applies to:

- All SOC analysts at Tier 1, Tier 2, and Tier 3
- All monitored environments including endpoint, network, cloud, and SaaS
- All incidents generated by Cortex XSIAM and integrated data sources
- All automation playbooks executing within the XSIAM Automation Engine

---

## 3. Roles and Responsibilities

| Role | Core Responsibilities |
|------|----------------------|
| Tier 1 Analyst | Initial triage, alert validation, false positive closure, case documentation, escalation to T2 when tier boundary is reached |
| Tier 2 Analyst | In-depth investigation, containment execution, tuning recommendations, case ownership on complex incidents, escalation to T3 |
| Tier 3 / SOC Lead | Complex incident handling, playbook governance, detection engineering oversight, leadership escalation, SLA accountability, quality control |
| Detection Engineer | Data source onboarding, rule development, MITRE ATT&CK alignment, tuning approval, ingestion health monitoring |

**Accountability Rule:** Every open case in XSIAM must have a named owner at all times. Zero unassigned incidents is the operational standard.

## 4. Incident Lifecycle

All incidents follow a five-phase response model. No phase may be skipped without documented justification recorded in the XSIAM case.

### Phase 1 — Detection and Validation

- XSIAM generates alert from integrated data source
- Analyst validates true positive vs. false positive within MTTD target
- False positives closed with documented rationale in case notes
- True positives triaged and severity confirmed or adjusted

**XSIAM Action:** Assign case, update status to In Progress, add initial triage note

### Phase 2 — Analysis and Investigation

- Full scope assessment performed: affected assets, user accounts, network activity, timeline
- MITRE ATT&CK technique mapped to the incident
- Context enriched using XSIAM threat intelligence integrations
- Severity confirmed or escalated based on investigation findings

**XSIAM Action:** Update case with investigation findings, attach evidence, confirm severity

### Phase 3 — Containment

- Isolate affected endpoints using Cortex XDR response actions
- Disable compromised accounts or block malicious IOCs in integrated controls
- Block C2 infrastructure through Palo Alto NGFW or Prisma Access integration
- All containment actions documented with timestamps in case notes

**XSIAM Action:** Execute response actions from XSIAM case, document each action taken

### Phase 4 — Eradication and Recovery

- Remove threat artifacts from affected systems
- Restore systems to known-good state
- Validate no persistence mechanisms remain through endpoint scan
- Confirm clean ingestion and normal telemetry post-remediation

**XSIAM Action:** Update case with remediation steps, confirm clean scan results

### Phase 5 — Post-Incident Review

- Document full incident timeline and root cause in case closure notes
- Capture lessons learned and detection gaps identified
- Submit tuning recommendations to prevent recurrence
- Close case in XSIAM with all required fields completed

**XSIAM Action:** Complete all closure fields, submit tuning ticket if applicable, close case

## 5. Severity and SLA Model

**SLA compliance target: 90% or above across all severity levels in all measurement periods.**

| Severity | MTTD Target | MTTA Target | MTTR Target | Escalation Path |
|----------|-------------|-------------|-------------|-----------------|
| Critical | 15 minutes | 15 minutes | 4 hours | T1 > T2 > T3 > Leadership |
| High | 30 minutes | 30 minutes | 8 hours | T1 > T2 > T3 |
| Medium | 1 hour | 1 hour | 24 hours | T1 > T2 |
| Low | 4 hours | 4 hours | 72 hours | T1 resolves |

### Operational Expectations

- Low and Medium incidents are resolved within the same shift whenever possible
- High incidents are resolved within the same business day
- Critical incidents receive immediate response with all-hands engagement if needed
- SLA breach cases are flagged in the XSIAM dashboard and reviewed same day

---

## 6. Escalation Path

### By Severity

- **Critical:** T1 acknowledges within 15 minutes, escalates to T2 immediately, T3 notified, leadership briefed
- **High:** T1 triages, escalates to T2 if not resolved within 2 hours
- **Medium:** T1 investigates, escalates to T2 if scope expands or resolution exceeds 8 hours
- **Low:** T1 resolves; escalates to T2 only if investigation reveals higher severity indicators

### Required Data at Escalation

- Incident ID and XSIAM case link
- Detection time and current severity
- Affected systems and user accounts
- Actions taken so far
- Recommended next step

## 7. Shift Operations

### Start of Shift

- Open XSIAM SLA Dashboard and review open queue
- Confirm all cases have named owners — pick up any unassigned immediately
- Identify SLA-at-risk cases and prioritize based on time remaining
- Review handoff notes from previous shift

### During Shift

- Maintain zero unassigned incidents at all times
- Escalate when tier boundaries are reached — do not hold cases
- Document all investigation actions in real time inside the XSIAM case
- Update case status as it progresses through the lifecycle

### End of Shift

- Ensure all active cases are assigned or closed
- Deliver handoff summary: open incidents, active escalations, SLA risks
- No case left without a documented owner or clear next action
- Log handoff in the shift notes section

## 8. Detection Engineering Governance

### Data Source Onboarding

1. Request submitted to SOC Lead with business justification
2. Relevance and coverage gap validated before integration begins
3. Integration configured and ingestion verified in XSIAM
4. Detections validated against test data before going live
5. Documentation completed: data source name, log type, expected volume, detection rules created

### Continuous Monitoring

- Monitor for log ingestion failures and parsing errors in XSIAM health dashboard
- Alert on volume anomalies that could indicate data loss or source failure
- Weekly ingestion health review by Detection Engineer

### Tuning Feedback Loop

- Every repeated alert without a tuning ticket is a process failure
- T2 analysts own tuning recommendations and submit via tuning register
- T3 approves all tuning changes before implementation
- Monthly detection review to retire stale or low-value rules

### MITRE ATT&CK Alignment

- All new detection rules mapped to MITRE ATT&CK technique on creation
- Coverage gaps reviewed quarterly against ATT&CK matrix
- High-priority techniques without coverage flagged to Detection Engineer

## 9. XSIAM Operational Standards

### Case Management

- All cases must have: severity, assigned analyst, status, and investigation notes
- Case notes updated at minimum at triage, investigation, containment, and closure
- Evidence and artifacts attached directly to the XSIAM case
- No case closed without root cause documented

### Dashboard Standards

- SLA Dashboard reviewed at start of every shift
- MTTD, MTTA, and MTTR metrics reviewed weekly by SOC Lead
- Analyst workload distribution reviewed for balance
- Overdue case rate tracked and kept below 5%

### Playbook Standards

- All playbooks version controlled and reviewed quarterly
- New playbooks require T3 sign-off before activation
- Playbook execution logs reviewed monthly for errors
- Automation failures escalated to Detection Engineer within 24 hours

### XQL Query Standards

- All operational XQL queries saved to the XSIAM query library
- Queries documented with purpose, data sources, and expected output
- Dashboard queries reviewed monthly to confirm data accuracy

## 10. Quality Control

- 48-hour quality control cycle on all closed incidents
- Random case sampling weekly for documentation completeness
- Tuning tickets required for any repeated alert type
- SLA breach post-mortems documented and reviewed at next shift briefing
- Cases failing QC review returned to analyst for remediation

**QC Checklist:**

- [ ] Severity appropriate for the activity observed
- [ ] All phases of the incident lifecycle documented
- [ ] MITRE ATT&CK technique mapped
- [ ] Evidence and artifacts attached
- [ ] Root cause documented
- [ ] Tuning recommendation submitted if applicable
- [ ] Case closed with all required fields complete

---

## 11. Performance Review

- Monthly SLA compliance review against 90% target
- MTTD, MTTA, and MTTR tracked per severity tier and per analyst
- Detection coverage reviewed against MITRE ATT&CK framework
- Analyst performance reviewed against tier expectations
- Tuning effectiveness measured by false positive rate trend
- Playbook automation success rate tracked monthly

**Monthly Reporting Inputs:**

| Metric | Source |
|--------|--------|
| SLA compliance rate | XSIAM SLA Dashboard |
| MTTD by severity | XSIAM XQL query |
| MTTR by severity | XSIAM XQL query |
| False positive rate | Case closure data |
| Tuning tickets submitted | Tuning register |
| Playbook execution rate | XSIAM automation logs |

## 12. Playbook Automation

### Case Notification Playbook

Built in the XSIAM Automation Engine to eliminate manual case routing and reduce acknowledge time.

**Triggers:**
- New case assigned to analyst
- Case approaching SLA breach threshold
- Case is overdue
- Daily analyst digest (scheduled)

**Actions:**
- Email notification sent to assigned analyst with case ID, severity, and SLA status
- Case comment added confirming notification was sent
- SOC Lead notified on Critical overdue cases

**Architecture:**
- `CaseNotification_Main` — parent playbook handling trigger routing
- `CaseNotification_SendEmail` — sub-playbook executing email delivery via dynamic `${incident.ownerEmail}` routing

### Automation Standards

- All playbooks imported and managed through XSIAM Incident Response > Automation > Playbooks
- Sub-playbooks imported before parent playbooks
- All automation tested in XSIAM Playground before production activation
- Execution logs reviewed monthly
